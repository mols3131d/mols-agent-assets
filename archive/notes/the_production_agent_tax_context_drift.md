# [에이전트 운영의 대가: Context Drift ≠ Hallucination](https://www.reddit.com/r/AI_Agents/comments/1riibyj/the_production_agent_tax_context_drift/)

> **이상과 현실**: 튜토리얼은 데모를 보여주지만, 실무 환경에서는 표류(Drift)가 발생합니다.
> 프로덕션 에이전트 운영 4개월간 겪은 실제 결함과 해결책을 정리했습니다.

---

## 1. 함정: 실패의 피드백 루프

1.  **테스트**: 테스트 환경에서는 완벽하게 동작하는 에이전트 구축.
2.  **배포**: 실무(Production) 환경에 배포.
3.  **저하**: 시간이 지날수록 점진적으로 신뢰성 하락.
4.  **오판**: Drift를 Hallucination(환각)으로 오해.
5.  **악순환**: 해결을 위해 프롬프트를 추가 (노이즈 및 Context Tax 증가).

---

## 2. 실제 발생하는 결함들

### A. Context Drift ≠ Hallucination

모델이 거짓말을 하는 것이 아니라, **State Anchor(상태 고정 장치)**가 없기 때문에 발생합니다.

- **문제**:
  - 에피소드 1: 파일 X를 읽고 결정 A 수행.
  - 에피소드 2: 파일 X를 다시 읽지만(컨텍스트 윈도우 변경), 결정 B 수행.
  - 에피소드 3: 파일 X를 이미 처리했다는 사실을 "망각".
- **원인**: 모델의 능력 문제가 아닌, 컨텍스트 관리의 실패.
- **해결**:
  - **Explicit State Anchors**: 휘발성 메모리가 아닌 파일(Brain 등)에 결정 사항을 명시적으로 기록.
  - **Idempotency(멱등성)**: 이미 처리된 작업은 실행 전 단계에서 명시적으로 건너뜀.

### B. 비동기 도구 호출: 소리 없는 실패

모델은 낙관적인 실행자입니다. 에러가 직접 반환되지 않는 한 성공했다고 가정합니다.

- **문제**:
  - 모델이 `tool_A` 호출.
  - `tool_A`가 느리거나 소리 없이 실패.
  - 모델은 `tool_A`가 성공한 것으로 간주하고 다음 단계 진행.
- **해결**:
  - **Synchronous Checkpoints**: 다음 턴을 시작하기 전, 반드시 상태 확인(Status Confirmation) 강제.
  - **Explicit Callbacks**: 도구의 에러를 다음 프롬프트 컨텍스트에 즉각 주입.

### C. 멀티-턴 플래닝: 누적되는 오류

세션이 길어질수록 모델은 "장기적인 맥락"을 놓치기 쉽습니다.

- **문제**:
  - 1턴: A -> B -> C 계획 수립.
  - 5턴: B를 다시 수행 (수행 이력 망각).
  - 8턴: C를 건너뜀 (원래 계획 소실).
- **해결**:
  - **Single-Turn Focus**: 한 턴에 하나의 태스크만 집중. 검증 후 종료.
  - **External Task Queues**: 계획을 모델의 메모리가 아닌 외부 파일(Kanban 등)에서 관리.

---

## 3. 실무 환경에서 작동하는 아키텍처

| 원칙                | 실행 방안                                               |
| :------------------ | :------------------------------------------------------ |
| **State Anchors**   | 영구 저장소에 결정 및 상태 기록 (비휘발성).             |
| **Hard Rails**      | 프롬프트가 아닌 코드(Validate) 수준에서 상태 전이 검증. |
| **Idempotency**     | 중복 실행이나 순환 오류를 방지하는 가드레일 구축.       |
| **Structured Logs** | 텍스트 로그 대신 타임라인 기반의 구조화된 이벤트 추적.  |

---

> [!IMPORTANT]
> **모델의 기억력을 믿지 마십시오. 명시된 상태(State)를 믿으십시오.**
> 운영의 핵심은 로직을 프롬프트가 아니라 인프라(Infrastructure)로 옮기는 것입니다.
