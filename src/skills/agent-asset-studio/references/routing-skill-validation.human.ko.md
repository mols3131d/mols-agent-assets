# 라우팅 스킬 검증 (Routing Skill Validation)

라우팅의 정확성, 구조적 안전성, 컨텍스트 효율성을 점검하기 위한 기준 가이드입니다.

## Expected Context (예상 컨텍스트 크기)

단일 경로 요청이 들어왔을 때 로드되는 리소스는 다음과 같아야 합니다:

```text
metadata + SKILL.md + INDEX.csv + 선택된 단일 워크플로우 + 필요한 리소스
```

다음과 같은 현상이 발생하면 라우팅 스킬의 효율성이 떨어지는 신호입니다:

- 인덱스 파일(`INDEX.csv`)이 너무 상세한 지침서나 매뉴얼처럼 비대해질 때
- 라우터(`SKILL.md`)가 개별 워크플로우의 세부 절차를 중복해서 적고 있을 때
- 대부분의 요청이 거의 모든 워크플로우 모듈을 동시에 로드하고 있을 때
- 라우터가 서로 연관 없는 도메인까지 무리하게 포괄하고 있을 때
- 워크플로우가 재귀적으로 다른 워크플로우를 호출(라우팅)할 때
- 정확한 파일 경로를 지정하는 대신 디렉토리 통째 스캔 방식을 쓸 때

일반적으로 항상 함께 로드되는 워크플로우들은 하나로 통합하십시오. 요청들이 단 하나의 경로로 깔끔하게 매칭되지 않는 상황이 빈번하다면 스킬을 분할하십시오.

## Structural Checks (구조적 점검 사항)

- 약속된 위치에 `INDEX.csv` 파일이 존재하는가
- 스키마가 정확히 `id,use_when,excludes` 형태로 구성되어 있는가
- 각 ID가 유일하고 안전한 상대 경로이며, 실제 스킬 내에 해당 파일이 존재하는가
- `use_when`과 `excludes`가 비어 있지 않으며, 의미상 명확히 구분되는가
- 리소스 경로들이 명시된 기준 경로로부터 올바르게 분석되는가
- 탐색을 위해 디렉토리 전체를 스캔하는 워크플로우가 없는가
- `workflows/` 디렉토리 하위에 탐색 가능한 독립 `SKILL.md`가 중첩되어 존재하지 않는가

## Routing Tests (라우팅 테스트 케이스)

최소한 다음 카테고리별로 각 1회 이상 테스트를 수행하십시오:

- 명확한 다이렉트 매칭 (Clear direct match)
- 간접적이거나 암시적인 매칭 (Indirect/implicit match)
- 특정 경로에서 제외 처리된 유사 매칭 (Near miss excluded)
- 명시적인 다중 워크플로우 요청 (Multi-workflow request)
- 도메인 범위를 벗어난 요청 (Out-of-scope request)
- 실질적인 모호함이 있는 요청 (Materially ambiguous request)

명확한 요청의 경우 최소한의 올바른 워크플로우 세트만 로드되는지, 범위 밖의 요청은 라우팅되지 않는지, 그리고 모호한 요청이 들어오면 정확한 질문을 통해 확인하는지 검증합니다.

## Measures (측정 지표)

올바른 경로 선택 비율, 오경로 지정 비율, 불필요한 워크플로우 로드 횟수, 질문 발생 빈도, 평균 로드 컨텍스트 크기를 지속적으로 모니터링합니다. 정확성(Correctness) 면에서의 하락은 실패로 취급합니다. 컨텍스트 절약을 핑계로 오라우팅을 정당화할 수 없습니다.
