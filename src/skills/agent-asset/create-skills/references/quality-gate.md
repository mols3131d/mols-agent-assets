# Agent Skills 품질 게이트

이 문서는 새 스킬 또는 고도화된 스킬을 최종 점검할 때 읽는다. 기준은
Agent Skills 공개 형식과 범용 클라이언트 호환성을 우선한다.

## 1. 형식 호환성

- 스킬 루트에 `SKILL.md`가 있다.
- frontmatter는 유효한 YAML이다.
- `name`은 폴더명과 일치한다.
- `name`은 1-64자이며 소문자 영문, 숫자, 하이픈만 포함한다.
- `description`은 1-1024자이며 기능과 사용 시점을 모두 설명한다.
- `compatibility`가 있으면 1-500자이며 실제 환경 요구사항만 적는다.
- `license`, `metadata`, `allowed-tools`는 필요할 때만 사용한다.

## 2. 트리거 품질

- `description`만 보고도 언제 사용할지 알 수 있다.
- 사용자의 자연어 표현이 3개 이상 포함된다.
- 너무 넓은 표현만 쓰지 않는다. 예: "개발", "문서", "자동화".
- 관련 없는 작업에서 잘못 호출될 가능성이 낮다.
- 활성화 테스트 쿼리를 `should_trigger=true/false`로 나누고 3회 이상 반복 평가한다.

## 3. 실행성

- 첫 화면에서 목표, 절차, 완료 기준이 보인다.
- 추상 원칙보다 행동 지시가 많다.
- "좋게 작성한다", "주의한다" 같은 표현은 구체 조건으로 바꾼다.
- 사용자가 요청하면 바로 파일을 만들거나 수정할 수 있을 만큼 절차가 분명하다.

## 4. 컨텍스트 절약

- `SKILL.md` 본문은 500줄 미만이다.
- 5000토큰을 넘길 가능성이 있으면 내용을 분리한다.
- 매번 필요하지 않은 긴 예시는 `references/`에 있다.
- 참조 파일은 `SKILL.md`에서 읽는 조건이 명확하다.
- 같은 내용이 `SKILL.md`와 참조 파일에 중복되지 않는다.
- 참조 파일은 한 단계 깊이에서 직접 연결한다.

## 5. 리소스 설계

- `scripts/`는 반복 실행, 결정적 처리, 복잡한 명령을 담을 때만 둔다.
- 스크립트는 비대화형이며 `--help`, 명확한 오류, 의미 있는 exit code를 제공한다.
- 데이터 출력은 stdout, 진단 메시지는 stderr로 분리한다.
- `references/`는 조건부로 읽을 상세 지식만 둔다.
- `assets/`는 실제 산출물에 복사하거나 삽입할 파일이 있을 때만 둔다.
- 특정 클라이언트 전용 설정은 범용 스킬의 기본 구성에 넣지 않는다.

## 6. 안전과 실패 처리

- 파괴적 작업, 외부 네트워크, 권한 상승이 필요한 경우 사용자 승인 조건을 적는다.
- 모호한 입력에서 질문해야 하는 기준이 있다.
- 상태 변경 작업은 idempotent하게 설계한다.
- 위험한 작업에는 `--dry-run`, `--confirm`, `--force` 같은 명시적 안전장치를 둔다.

## 7. 평가와 반복

- 실제 사용 요청을 train/validation 예시로 나눈다.
- 스킬이 없는 기준 실행과 스킬을 사용한 실행을 비교한다.
- 성공 출력, 실패 출력, 실행 trace를 모두 수정 근거로 삼는다.
- description을 여러 번 개선하되 validation 통과율이 가장 높은 버전을 선택한다.
- `skills-ref validate <skill-dir>`와 로컬 보조 검증을 모두 통과시키는 것을 목표로 한다.
