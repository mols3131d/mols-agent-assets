# 에이전트 자산의 핵심 원칙 (The Zen of Agent Assets)

에이전트 자산은 단순한 추가 지침서가 아닙니다. 더 작은 컨텍스트(Context)로 더 정확한 작업을 수행할 수 있도록 돕는 인터페이스(Interface)입니다.

1. 장황함보다 간결함이 낫습니다. (Concise beats verbose.)
2. 비결정론보다 결정론적인 것이 낫습니다. (Deterministic beats nondeterministic.)
3. 암시적인 것보다 명시적인 것이 낫습니다. (Explicit beats implicit.)
4. 여러 개의 책임보다 하나의 명확한 작업이 낫습니다. (One clear job beats many duties.)
5. 디렉토리 검색보다 정확한 경로 지정이 낫습니다. (Exact paths beat directory scans.)
6. 지문을 반복하는 것보다 실행 가능한 스크립트가 낫습니다. (Executable scripts beat repeated prose.)
7. 규칙을 복제하는 것보다 단 하나의 진실 공급원(Single Source of Truth)을 두는 것이 낫습니다. (One source of truth beats duplicate rules.)
8. 모든 것을 한 번에 로드하는 것보다 필요한 때 로드(On-demand)하는 것이 낫습니다. (On-demand loading beats loading everything.)
9. 키워드 매칭보다 사용자의 실제 의도가 중요합니다. (User intent beats keyword matching.)
10. 기발한 예외 상황 처리보다 단순한 기본값 설정이 낫습니다. (Boring defaults beat clever exceptions.)
11. 근거 없는 확신보다 검증 가능한 결과가 중요합니다. (Verifiable results beat unsupported confidence.)
12. 대대적인 개편보다 기존 동작을 보존하는 최소한의 변경이 낫습니다. (Small behavior-preserving changes beat rewrites.)
13. 파괴적인 변경보다 원상복구가 가능한 변경이 낫습니다. (Reversible changes beat destructive ones.)
14. 추측보다 눈에 보이는 증거가 중요합니다. (Observable evidence beats guesses.)
15. 비어 있는 향후 뼈대(Scaffolds)보다 실제로 당장 필요한 자산을 만드는 것이 낫습니다. (Needed assets beat empty future scaffolds.)
16. 아무 말 없이 오류를 묵인하는 것보다 명확한 중단 조건을 두는 것이 낫습니다. (Clear stop conditions beat silent errors.)
17. 발생 가능한 모든 조합을 상정하는 것보다 필요최소한의 워크플로우를 두는 것이 낫습니다. (Minimum sufficient workflows beat every possible combination.)
18. 독창적인 방식보다 공유된 관례(Conventions)를 따르는 것이 낫습니다. (Shared conventions beat novel ones.)

간결함을 유지해야 핵심 의미가 보존됩니다. 결정론적이어야 올바른 판단을 내릴 수 있습니다. 단순함을 추구하되 안전성, 정확성 및 사용 편의성(Accessibility)을 절대 타협하지 마십시오.

규칙들이 서로 충돌할 때는 사용자 의도(User intent), 안전(Safety), 검증 가능성(Verifiability)이 항상 우선합니다. 예외 사항은 명시적으로 기록하고, 그 이유와 경계를 문서로 남기십시오.
