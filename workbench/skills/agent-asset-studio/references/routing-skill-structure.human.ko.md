# 라우팅 스킬 구조 아키텍처 (Routing Skill Architecture)

최소한의 컨텍스트만 로드하면서 요청을 작업별 워크플로우로 전달하는 하나의 통합 라우팅 스킬을 구축하기 위한 참조 사양입니다.

> **Status (상태):** 라우팅 스킬은 에이전트 스킬(Agent Skills) 스펙을 바탕으로 구현된 사용자 정의 패턴입니다. `INDEX.csv`와 `workflows/`는 자체적으로 정한 관례(Conventions)이며, 에이전트 스킬 표준 기본 사양(Primitives)은 아닙니다.

## 1. Purpose and Fit (목적 및 적합성)

라우팅 스킬은 하나의 일관된 도메인 능력을 외부에 제공하며, 현재 요청에 필요한 특정 워크플로우만 선택적으로 로드합니다.

장점:

- 좁은 범위의 여러 스킬들 대신 단 하나의 카탈로그 엔트리만 유지
- 필요한 워크플로우만 선택적으로 로드하여 컨텍스트 절약
- 정책, 스크립트, 레퍼런스, 에셋 등의 리소스를 공유
- 명확한 라우팅 경계 형성
- 중앙 집중식 유지보수 용이

워크플로우들이 하나의 도메인, 공유 리소스, 최상위 활성화 설명을 공유하는 경우에 라우팅 스킬을 사용하십시오. 스킬들의 트리거, 권한, 소유권 또는 릴리즈 라이프사이클이 서로 무관한 경우에는 개별 스킬로 분리하여 빌드하십시오.

| 상황 (Situation) | 설계 방식 (Design) |
| --- | --- |
| 단일 도메인 내의 여러 변형 작업들 | 라우팅 스킬 적용 |
| 워크플로우들이 자주 서로 엮이거나 구성될 때 | 라우팅 스킬 적용 |
| 서로 연관 없는 의도나 권한 경계를 가질 때 | 개별 스킬 분리 |
| 몇 개의 짧은 워크플로우만 있는 경우 | 단일 기본 `SKILL.md` 구성 |

가장 주된 최적화 목표는 파일시스템 스캔 비용이 아니라, 에이전트 모델의 카탈로그 크기와 컨텍스트 창(Context window) 사용량을 최소화하는 것입니다.

## 2. Recommended Structure (권장 구조)

```text
routing-skill/
├── SKILL.md          # 활성화 메타데이터 및 라우터 (Router)
├── workflows/        # 라우팅 대상 워크플로우 모듈들
│   ├── INDEX.csv     # 라우팅 경로 등록 인덱스 (기본 위치)
│   ├── ingest-data.md
│   ├── transform-data.md
│   └── validate-quality.md
├── scripts/          # 재사용 가능한 실행 로직 (스크립트)
├── references/       # 공유 지식 및 참고 명세서
└── assets/           # 템플릿 및 정적 리소스
```

용어 정리:

- 전체 패키지: **라우팅 스킬 (Routing Skill)** 또는 **복합 스킬 (Composite Skill)**
- 루트 `SKILL.md`: **스킬 라우터 (Skill Router)**
- `INDEX.csv`: **경로 인덱스 (Route Index)**
- 각 마크다운 파일: **워크플로우 모듈 (Workflow Modules)**

개별적으로 탐색되지 않는 하위 모듈에 대해서는 "서브 스킬(sub-skill)"이라는 표현 대신 "워크플로우 모듈"이라는 용어를 사용해 혼선을 방지하십시오.

## 3. `SKILL.md`: The Router (스킬 라우터)

`SKILL.md`는 스킬의 유일한 진입점입니다. 매 활성화마다 로드되므로 크기를 작게 유지해야 합니다.

포함할 항목:

- 일관된 도메인 범위를 기술하는 프론트매터(Frontmatter)
- 라우팅 및 모호성 해결 규칙
- 전역 제약 사항
- 공유 리소스 로드 조건

워크플로우의 구체적인 전체 절차, 방대한 레퍼런스 내용, 디렉토리 내의 모든 모듈을 강제 스캔하도록 지시하는 규칙 등은 포함하지 마십시오.

## 4. `INDEX.csv`: The Route Index (경로 인덱스)

라우팅 스킬은 기본적으로 `workflows/INDEX.csv` 경로에 있는 인덱스 파일을 사용합니다. 필드는 `id,use_when,excludes`로 구성되며, 각 `id` 경로는 인덱스 파일이 위치한 디렉토리를 기준으로 상대 해석합니다.

인덱스를 생성하거나 업데이트할 때는 `workflows/agent-asset-index-write.md` 워크플로우를 사용하십시오. 인덱스 포맷 작성 규칙을 라우터 파일에 중복해서 기록하지 마십시오.

## 5. Workflow Modules (워크플로우 모듈)

각 모듈은 특정 작업의 진입점(오케스트레이터) 역할을 하며, 단 하나의 실행 가능한 워크플로우를 담고 다음의 일관된 계약(Contract) 형식을 따릅니다:

```markdown
# 워크플로우 이름 (Workflow Name)

## Goal (목표)
달성하고자 하는 결과물을 기술합니다.

## Required inputs (필수 입력값)
실행 전에 확보해야 하는 입력 정보 목록입니다.

## Procedure (절차)
간결하게 정렬된 실행 단계를 제공합니다.

## Validation (검증)
작업이 완료되었는지 확인하는 체크 리스트입니다.

## Resources (참고 리소스)
참조 문서를 로드하거나 스크립트를 실행해야 하는 시점을 밝힙니다.

## Stop conditions (중단 조건)
사용자에게 질문을 던지거나 작업을 즉시 멈춰야 하는 조건을 기술합니다.
```

`SKILL.md`에 기술된 전역 규칙을 워크플로우 내에 중복 기재하지 마십시오. 단순 지식 정보는 `references/`에, 반복되는 결정론적 로직은 `scripts/`에, 템플릿 파일은 `assets/`에 각각 격리하여 배치합니다.

## 6. Recommended Defaults (권장 기본값)

| 의사결정 항목 | 권장 기본값 |
| --- | --- |
| 모듈 디렉토리 위치 | `workflows/` |
| 워크플로우 파일 경로 | `workflows/<workflow-name>.md` |
| 리소스 로드 방식 | 명시적 로드 및 필요할 때만 로드 (On-demand) |
