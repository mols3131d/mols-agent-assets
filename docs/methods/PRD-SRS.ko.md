# PRD 및 SRS 가이드

제품의 가치를 정의하는 비즈니스 언어를 개발이 가능한 기술 언어로 번환하는 과정은 프로젝트 성공의 핵심입니다. 본 가이드 PRD와 SRS의 정의와 유기적인 연동 방식을 설명합니다.

---

## 1. PRD (Product Requirements Document)

PRD는 제품의 목적과 방향성을 정의하는 문서로, 제품이 **'무엇(What)'**이어야 하고 **'왜(Why)'** 만들어야 하는지에 집중합니다. 주로 제품 매니저(PM)가 주도하여 작성합니다.

- **대상**: 이해관계자, 디자이너, 개발 팀 전체.
- **주요 내용**: 비즈니스 목표, 사용자 페르소나, 사용자 스토리, 성공 지표(KPI), 상위 수준의 요구사항.
- **특징**: 기술적인 구현 방법보다는 사용자의 가치와 비즈니스 논리에 중점을 둡니다.

## 2. SRS (Software Requirements Specification)

SRS는 PRD의 요구사항을 어떻게 **'기술적으로 구현(How)'**할 것인지 상세히 명시한 문서입니다. 시스템 분석가나 리드 개발자가 작성합니다.

- **대상**: 개발자, QA 엔지니어, 유지보수 팀.
- **주요 내용**: 데이터 모델, API 명세, 시스템 아키텍처, 성능 요구사항, 제약 사항, 상세 유스케이스.
- **특징**: 매우 구체적이고 정교하며, 실제 구현과 검증(QA)의 절대적인 기준점이 됩니다.

---

## 3. 요구사항 변환 프로세스

비즈니스 요구가 기술적 명세로 변환되어 실제 제품으로 구현되는 단계별 흐름입니다.

1.  **시장 분석 및 기획**: 사용자의 불편함을 파악하고 이를 해결하기 위한 솔루션을 구상합니다.
2.  **PRD 작성**: PM이 제품의 목적과 필요한 기능 목록을 비즈니스 관점에서 정리하여 공유합니다.
3.  **기술 검토 및 분석**: 개발 팀이 PRD를 검토하며 기술적 실현 가능성을 확인하고 세부 설계를 시작합니다.
4.  **SRS 작성**: PRD의 각 기능을 구현하기 위한 데이터베이스 구조, 로직, 인터페이스 사양을 기술적으로 상세히 기록합니다.
5.  **개발 및 검증**: 개발자는 SRS를 기준으로 코드를 구현하며, QA는 SRS의 명세에 따라 정상 작동 여부를 검증합니다.
