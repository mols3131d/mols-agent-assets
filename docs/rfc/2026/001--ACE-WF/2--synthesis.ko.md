# 합(Synthesis): 피드백 중심의 순환 워크플로우

**정(0)**의 구조적 거버넌스와 **반(1)**이 요구하는 운영적 민첩성을 조화시키기 위해, `ACE-WF`를 선형 순서에서 반복 가능한 피드백 루프로 재정의합니다.

## 1. 핵심 변화: 단계(Phase)에서 상태(State)로

5개의 순차적 단계를 지양하고, 내부 반복이 가능한 3가지 주요 상태를 정의합니다.

### 워크플로우 구조

1. **Alignment (정렬)**: PDD와 REQ가 상호 작용하며 논리의 뼈대를 만듭니다.
2. **Evolution (진화)**: 제작(Studio)과 지속적 평가(CE)가 실시간으로 교차하며 품질을 높입니다.
3. **Maturity (성숙)**: 검증된 자산이 에셋으로 확정되어 배포 및 정리됩니다.

## 2. 주요 조율점: 기술 공학에서 '경험 공학'으로의 전환

### A. "드래프트 바인딩" (인지적 유연성 확보)

- **해결**: 요구사항 확정이 제작을 가로막지 않도록 `Draft-REQ` 체계를 유지합니다. 논리는 제작(Studio) 과정에서 발견되는 것이며, 문서화는 그 발견을 기록하는 과정입니다.

### B. "경험적 검증" (객관적 테스트의 한계 인정)

- **통찰**: 에이전트 지능은 소프트웨어처럼 객관적으로 테스트할 수 없으며, 자동화된 평가는 오히려 위험할 수 있습니다.
- **해결**: Phase 4(평가)를 **주관적/경험적 검증**으로 대체합니다.
  - **검증 기준**: "실제 다른 프로젝트나 세션에서 제대로 작동하는가?"
  - **피드백 루프**: 사용자가 직접 사용해보고 마음에 들지 않으면 언제든 Forge(제작) 단계로 회귀합니다.

### C. "재사용성이 곧 품질"

- 자산의 완성도는 문서상의 명세 충족이 아니라, **사용자의 작업 흐름에 얼마나 자연스럽게 녹아드는가**와 **재사용 시 의도가 유지되는가**로 판단합니다.

## 3. 개정된 운영 모델

| 항목            | 기존 논리 (정/반)         | 개정된 합 (경험 중심)                    |
| :-------------- | :------------------------ | :--------------------------------------- |
| **검증 방식**   | 자동화 테스트 / 로그 감사 | **사용자 주관적 승인 / 실전 재사용**     |
| **실패의 정의** | 테스트 케이스 통과 실패   | **"알아먹지 못함" / "실전 적용 불가"**   |
| **완료 기준**   | 요구사항 명세 일치        | **실제 세션에서의 논리 정렬(Alignment)** |

---

## 결론

본 합(Synthesis)안은 자산 관리에 필요한 **통제력**을 잃지 않으면서도, 지능 엔지니어링에 필수적인 **속도**를 확보합니다. 기존의 5단계 모델은 개념적으로 보존하되, 실행 면에서는 재귀적인 루프로 작동하게 됩니다.
