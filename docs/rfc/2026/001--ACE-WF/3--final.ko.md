# 결정: RFC-ADR 기반의 ACE 워크플로우

레니(정), 카나(반), 린(합) 사이의 토론 결과, `mols-agent` 프로젝트의 지능 자산 엔지니어링을 위해 **통합 RFC-ADR 라이프사이클**을 공식 워크플로우로 채택합니다.

## 1. 핵심 라이프사이클

워크플로우는 선형적인 상하달식 구조에서 벗어나, 의사결정 기록(ADR)과 실전 재사용을 중심으로 한 경험 주관적 루프로 전환됩니다.

1.  **정렬 (제안 및 RFC)**:
    - 모든 새로운 로직이나 변경 사항은 **제안(Proposal)**에서 시작합니다.
    - **RFC (Request for Comments)** 단계를 거쳐 페르소나들(레니, 카나, 린)이 로직을 비판하고 정교화합니다.
2.  **결정 (ADR 및 요구사항)**:
    - RFC의 결론은 **ADR (Architecture Decision Record)**로 명문화됩니다.
    - ADR을 바탕으로 공식 **요구사항(Requirements)**이 작성됩니다.
3.  **제작 (Studio 및 통찰)**:
    - **Studio** 환경에서 실제 구현이 이루어집니다.
    - 제작 과정에서 발견되는 새로운 논리는 실시간으로 요구사항에 반영됩니다 (Draft 단계 활용).
4.  **검증 (주관적 승인 및 자산화)**:
    - 사용자의 주관적 만족과 실전 적용 성공 후에만 **Outputs/Assets**로 이동합니다.
    - **최종 기준**: 해당 자산을 다른 프로젝트에 다시 들고 갔을 때 의도대로 작동하는지가 품질의 척도가 됩니다.

## 2. 핵심 원칙

- **객관성보다 주관적 효용**: 컨텍스트 엔지니어링은 소프트웨어와 달라 객관적 수치화가 불가능함을 인정하며, 최종 권위는 사용자의 주관적 판단과 재사용 결과에 둡니다.
- **추적성(Traceability)**: ADR은 로직이 왜 존재하는지에 대한 영구 기록 역할을 하며 지식 격차를 방지합니다.
- **유연한 드래프트**: 공식 문서화가 제작 속도를 늦추지 않도록 **드래프트 요구사항** 상태에서의 선제적 제작을 허용합니다.
